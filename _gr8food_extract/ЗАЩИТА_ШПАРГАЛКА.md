## 2. Как работает программа (поток выполнения)

```
Program.cs  →  запускает окно LoginForm (вход)
```

## 3. ООП в проекте — как объяснять (ВАЖНО)

Если препод спросит «где у вас ООП», отвечаем тем, что **реально проходили в Lab 7**:

### Классы и объекты
В системе есть классы данных: `User`, `MenuItem`, `Order`, `OrderItem`, `Feedback`, `WalletTransaction`. Мы создаём объекты через `new`, например:
```csharp
User user = new User(userID, lid, role, name, email, phone);
```

### Свойства (Properties)
Данные в классе хранятся в свойствах с `{ get; set; }`:
```csharp
public string FullName { get; set; }
public decimal WalletBalance { get; set; }
```

### Перегруженные конструкторы (Overloaded constructors) — как в Lab 7
У класса `User` **два конструктора**: пустой и с параметрами:
```csharp
public User() { ... }                                  // пустой
public User(int userID, string loginID, string role, ...) { ... }   // с данными
```

### Методы класса с логикой — как `Deposit`/`Withdraw` в Lab 7 (Account)
У `User` есть методы для кошелька:
```csharp
public bool CanAfford(decimal amount) { return WalletBalance >= amount; }
public void Deduct(decimal amount)    { WalletBalance = WalletBalance - amount; }
public void Refund(decimal amount)    { WalletBalance = WalletBalance + amount; }
```
У `Order` — методы `CanBeCancelled()`, `CanLeaveFeedback()`. У `Feedback` — `GetRatingStars()`.

### Инкапсуляция (Encapsulation)
Данные и методы, которые с ними работают, лежат вместе в одном классе. Кошелёк меняется только через методы `Deduct`/`Refund`, а не напрямую.

> **Если спросят про наследование честно:** «Наследование мы не использовали — в задании оно не требовалось, и мы решили сделать как проходили в Lab 7: один класс `User` с полем `Role`, которое определяет роль. Это проще и нам понятнее.» Это нормальный, честный ответ.

## 5. Что делает каждая роль (своими словами)

- **Admin** — заводит/редактирует/деактивирует пользователей, смотрит отчёт о продажах.
- **Chef** — управляет своим меню (CRUD блюд), меняет статус заказов: Pending → In Progress → Completed.
- **Manager** — читает отзывы и отвечает на них, смотрит отчёт по операциям с кошельками.
- **Customer** — смотрит меню, складывает в корзину, оформляет заказ с оплатой из кошелька, пополняет кошелёк, смотрит историю, оставляет отзыв.

## 6. База данных (7 таблиц)

```
Users ──┬── Customers      (только у клиентов есть кошелёк)
        ├── MenuItems      (ChefID → какой повар добавил блюдо)
        └── Orders ──┬── OrderItems
                     ├── Feedback
                     └── WalletTransactions
```

- `Users` — общая таблица для ВСЕХ ролей; колонка `Role` отличает их; `IsActive` — активен ли (деактивация вместо удаления).
- `Customers` — баланс кошелька и адрес только для клиентов; `CustomerID` ссылается на `Users.UserID`.
- Связи — через внешние ключи (FOREIGN KEY). Ограничения — через CHECK (например, `Rating BETWEEN 1 AND 5`).

**Тестовые аккаунты:**
| Логин | Пароль | Роль |
|---|---|---|
| `admin001` | `Admin@123` | Admin |
| `chef001` | `Chef@123` | Chef |
| `manager001` | `Mgr@123` | Manager |
| `cust001` | `Cust@123` | Customer (баланс RM 50) |
| `cust002` | `Cust@123` | Customer (баланс RM 30) |

## 7. Как работаем с базой данных (как в Lab 8)

Вся работа с БД идёт через класс `DBHelper` тремя методами:
- `ExecuteQuery(...)` — для SELECT, возвращает таблицу данных (`DataTable`).
- `ExecuteNonQuery(...)` — для INSERT / UPDATE / DELETE.
- `ExecuteScalar(...)` — для одного значения (COUNT, MAX, новый ID).

Везде используем **параметры** (`@L`, `@P`...) — как в Lab 8:
```csharp
"WHERE LoginID=@L AND Password=@P"
new SqlParameter("@L", loginID)
```
**Зачем параметры:** безопасность, защита от SQL-инъекций (когда в поле ввода пишут вредный SQL).

**Пример: оформление заказа** (`MenuBrowseForm`) — делаем по шагам, один запрос за другим:
1. добавить заказ в `Orders` и получить его новый ID (`SCOPE_IDENTITY`),
2. в цикле `foreach` добавить позиции в `OrderItems`,
3. списать деньги: `UPDATE Customers SET WalletBalance = WalletBalance - @Amt`,
4. записать операцию в `WalletTransactions`.

Перед этим — проверка `cust.CanAfford(total)`: хватает ли денег.

## 9. Вероятные вопросы преподавателя + ответы

**В: Где у вас ООП / классы?**
О: У нас есть классы данных (`User`, `MenuItem`, `Order`...). Создаём объекты через `new`, у них свойства `{get;set;}`, перегруженные конструкторы и методы с логикой — как в Lab 7 (классы Car, Employee, Account).

**В: Вы использовали наследование?**
О: Нет, по заданию оно не требовалось. Мы сделали как проходили: один класс `User` с полем `Role`. Это проще и понятнее.

**В: Покажи перегруженный конструктор.**
О: В классе `User` два конструктора — пустой `User()` и `User(int, string, string, ...)` с данными. Как у класса Employee в Lab 7.

**В: Где у вас методы класса (не формы)?**
О: В `User` — `CanAfford`, `Deduct`, `Refund` (работа с кошельком, как Deposit/Withdraw в Lab 7). В `Order` — `CanBeCancelled`. В `Feedback` — `GetRatingStars`.

**В: Как храните пароли?**
О: Сейчас как обычный текст — для простоты учебного проекта. В реальной системе их нужно хешировать (например SHA-256 с солью).

**В: Как работаете с базой данных?**
О: Через ADO.NET — `SqlConnection`, `SqlCommand`, параметры (`@`), методы `ExecuteQuery/ExecuteNonQuery/ExecuteScalar`. Чтобы не повторять код подключения, вынесли его в класс `DBHelper`.

**В: Зачем класс DBHelper?**
О: Чтобы не писать `new SqlConnection`, `con.Open()` в каждой форме. Один раз написали — везде используем (меньше повторов).

**В: Как разграничены права ролей?**
О: После входа по роли открывается только её дашборд с её кнопками. У повара нет кнопки «Manage Users».

**В: Чем ExecuteQuery отличается от ExecuteNonQuery?**
О: `ExecuteQuery` — для SELECT, возвращает данные (`DataTable`). `ExecuteNonQuery` — для INSERT/UPDATE/DELETE. `ExecuteScalar` — одно значение.

**В: Почему пользователь не удаляется, а деактивируется?**
О: `UPDATE Users SET IsActive=0` — «мягкое удаление», чтобы не потерять связанные заказы. При входе проверяем `IsActive=1`.

**В: Где у вас цикл / выборка (Lab 3, 4)?**
О: `if/else` и `switch` при определении роли и проверках; `foreach` — при подсчёте корзины и добавлении позиций заказа.
